# Sync Pull Request

> 이 커맨드는 커밋을 만들지 않는다. 미완성 작업이 있다면 먼저 `/commit` 으로 커밋·push를 완료한 뒤 실행한다.

## 실행 전 중단 조건

아래 상태 중 하나라도 해당하면 이유를 설명하고 즉시 멈춘다.

| 상태             | 판단 기준                          | 안내 메시지                            |
| ---------------- | ---------------------------------- | -------------------------------------- |
| detached HEAD    | 현재 브랜치명이 비어 있음          | `git checkout -b <브랜치명>` 후 재실행 |
| merge conflict   | staged 파일 중 unmerged 항목 존재  | 충돌 해소 후 재실행                    |
| upstream 없음    | remote 추적 브랜치가 설정되지 않음 | `git push -u origin HEAD` 후 재실행    |
| 미push 커밋 존재 | 로컬이 원격보다 ahead              | `/commit` 또는 `git push` 후 재실행    |

---

## 1. 컨텍스트 파악

아래 항목을 모두 파악한다.

| 항목              | 파악 방법                                                                                   |
| ----------------- | ------------------------------------------------------------------------------------------- |
| 현재 브랜치명     | git에서 확인                                                                                |
| 원격 동기화 상태  | git status로 확인                                                                           |
| 이슈 번호         | 브랜치명에서 `[A-Z]+-\d+` 패턴을 우선 추출. 없으면 마지막 segment. 그것도 없으면 `NO-ISSUE` |
| Jira URL          | `https://jira.mailplug.co.kr/browse/<이슈번호>`                                             |
| develop 이후 커밋 | `origin/develop..HEAD` 범위의 커밋 subject와 body                                           |
| 변경 파일 목록    | `origin/develop` 대비 변경된 파일과 상태                                                    |
| 기존 열린 PR 여부 | 현재 브랜치를 head로 하는 open 상태 PR이 있는지 확인                                        |

---

## 2. PR 제목 결정

기존 PR이 이미 열려 있으면 이 단계를 건너뛴다. **제목은 절대 변경하지 않는다.**

우선순위대로 적용한다.

1. 사용자가 자연어로 제목을 직접 지정한 경우 → 그대로 사용
2. develop 이후 커밋이 1개 → 해당 커밋 subject를 그대로 사용
3. develop 이후 커밋이 2개 이상 → 핵심 내용을 하나의 대표 제목으로 재구성. Type이 모두 같으면 그대로 유지, 다르면 `Chore` 사용
4. 커밋이 없으면 → `Chore(<이슈번호>): 작업 반영`

제목 규칙은 다음을 따른다.

- 최대 50자
- 마침표·특수기호 없음
- `Type(scope): 설명` 형식 준수

## 3. PR 본문 생성

`.github/pull_request_template.md` 를 기반으로 작성한다. placeholder를 실제 값으로 치환하고, 각 섹션을 아래 기준으로 채운다.

### Related Issue

```md
## Related Issue 🔗

- Jira: [<이슈번호>](<Jira URL>)
```

### Summary

리뷰어가 코드를 열기 전에 왜 이 PR이 필요한지, 무엇이 바뀌었는지 이해할 수 있어야 한다. 파일명 나열 금지. 기능·동작 단위로 서술한다.

변경의 의도를 드러내는 핵심 코드가 있으면 코드 블록으로 첨부한다. 전체 diff가 아닌 핵심 부분만 발췌한다.

````md
## Summary ✏️

### 변경 배경

<!-- 왜 이 작업이 필요했는가? 기존의 문제나 요구사항을 한두 문장으로 -->

### 핵심 변경사항

<!-- 무엇이 바뀌었는가? 파일명 나열 금지. 기능·동작·구조 단위로 설명 -->

<!-- 변경의 의도를 드러내는 핵심 코드가 있으면 아래에 첨부 -->

```ts
// 변경의 의도를 드러내는 핵심 코드 (해당 시)
```

### 기대 효과

<!-- 이 변경으로 무엇이 나아지는가? 정량적이면 더 좋음 -->
````

### 작성 규칙

| 규칙                  | 이유                                                                                              |
| --------------------- | ------------------------------------------------------------------------------------------------- |
| 파일명·경로 나열 금지 | 리뷰어는 diff로 확인 가능. 본문은 의도를 전달해야 함                                              |
| 기능·동작 단위로 설명 | 파일명이 아닌 변경된 동작과 그 이유를 중심으로 서술                                               |
| 핵심 코드 스니펫 첨부 | 변경의 의도를 한눈에 드러내는 코드가 있으면 코드 블록으로 삽입. 전체 diff가 아닌 핵심 부분만 발췌 |
| 리뷰 포인트 명시      | 설계 결정이나 트레이드오프가 있으면 별도 언급                                                     |
| 영향 범위 명시        | 다른 모듈·페이지에 사이드이펙트가 있으면 반드시 기재                                              |

## 4. PR 생성 또는 업데이트

| 상황         | 행동                                                |
| ------------ | --------------------------------------------------- |
| 열린 PR 없음 | base `develop` 으로 새 PR 생성                      |
| 열린 PR 있음 | 본문만 업데이트. 제목·라벨·assignee는 건드리지 않음 |

---

## 5. 후처리

현재 GitHub 사용자를 확인한 뒤 아래를 수행한다. 이미 설정된 값은 덮어쓰지 않는다.

1. `cursor-generated` 라벨이 없으면 생성 후 PR에 추가
2. 현재 사용자를 assignee로 추가
3. `.github/CODEOWNERS` 가 있으면 reviewer를 추출해 본인 제외 후 추가

## 6. 결과 출력

```
Action  : created | updated
Branch  : <브랜치명>
PR      : <URL>
Number  : #<번호>
Title   : <제목>
Body    : updated
```

## 실패 처리

| 시점                   | 처리 방법                                     |
| ---------------------- | --------------------------------------------- |
| 실행 전 중단 조건 해당 | 현재 상태 설명 + 해소 방법 안내 후 중단       |
| upstream 없음          | `git push -u origin HEAD` 안내 후 중단        |
| 미push 커밋 존재       | `/commit` 또는 `git push` 안내 후 중단        |
| PR 생성 실패           | 현재 브랜치명 + 마지막 커밋 제목 출력 후 중단 |
| PR 업데이트 실패       | 기존 PR 번호 / URL + 실패 원인 출력 후 중단   |
